## Issues

### P1 — Nested lists: new numbering instance **and** elevated `ilvl`

Each nested `ul`/`ol` allocates a **new** `reference` / abstractNum, then sets  
`currentListLevel = parentLevel + 1`.

Effects:

1. Nested **`ol`** at depth 1 uses template **level 1** = `LOWER_LETTER` + `%2.`, not `DECIMAL` + `%1.`. Markers look like `a.` / `b.` (or odd `%2` parent linkage) rather than `1.` / `2.` under the parent item.
2. Because the nested instance never emits level-0 items, multi-level placeholders (`%2`, `%3`) do not reflect the **outer** list’s counter (different `numId`).
3. Deep mixed nests work enough to pass “ilvl 0/1/2 exists” tests but are not Word-native “one abstract, many levels” nesting.

**Preferred fix (Stage 8 or quick follow-up — not Claude Stage 5):**

- **Same list type nest:** reuse `parentRef`, only increment `ilvl` (single abstract, true Word nesting).  
- **Type switch (ul↔ol):** new reference but force **`ilvl = 0`** and apply extra `indent.left` from depth so markers restart at 1./• with visual nest.  
- Add fixture assert: nested `ol` items use `numFmt` decimal (or explicit accepted letter scheme documented in BUILD_PLAN deviations).

**Does not block Stage 5** (tables).

### P2 — First-`p` hoist drops the `p`’s own computed style

`getListItemContent` pulls `p`’s children into `inlineNodes` and styles them with **`li`’s** `resolveRunProps`, not the `p`’s CSS.  
`<li><p class="lead">…</p>` may lose paragraph-level color/weight.

### P2 — Non-`p`/`div` blocks inside `li`

`h1`–`h6`, `pre`, `blockquote`, `table` (soon) classified as “inline” in the else-branch → forced through `convertInline` or wrong path. Rare in clean HTML; Stage 8: treat known block tags as `blockNodes`.

### P2 — Tests thin on nest quality

Current tests prove numbering.xml exists, ilvl 0/1/2, restart numIds, inline color. They do **not** assert nested `ol` marker format or continuation indent. Strengthen when fixing P1.

### P3 — HANDOFF claims “perfect” visual

Accept as owner signal; Janus did not re-open PDF. Marker-format P1 may still “look fine” for bullets more than for nested decimals.

### P3 — `/mnt/c/temp` dependency
